# 2.5DC (5 voters across 3 regions, 2-2-1 split)

> See [`README.md`](README.md) for the `alive/needed/configured` notation, and [`FAULT_TOLERANCE.md`](../FAULT_TOLERANCE.md) for the underlying KRaft fault-tolerance concepts this guide applies.

This repo's example. Two Kafka-bearing regions plus a small "tiebreaker" region with one KRaft-only voter.

> **Terminology** (per [Confluent's Multi-DC Architectures doc](https://docs.confluent.io/platform/current/multi-dc-deployments/multi-region-architectures.html) and CFK design docs): **"2.5DC" specifically means two full DCs with Kafka brokers + KRaft controllers, plus a third "0.5"/tiebreaker DC that hosts only KRaft controllers (no Kafka brokers)**. The defining feature is the broker-less third region — not any particular voter split. A deployment with symmetric KRaft across 3 DCs (e.g. 2-2-2) where Kafka brokers happen to be in only 2 of those DCs is *not* 2.5DC; it's a 3DC KRaft cluster with asymmetric Kafka placement, and the math/DR shape follows 3DC, not 2.5DC. Confluent also uses **"stretch cluster" / "stretched cluster"**, **"0.5DC"**, **"tiebreaker site"**, and **"light DC"** for the same topology. **"2+1"** and **"MRC with tiebreaker"** are not formal Confluent terms.

**Recommended voter split: 2-2-1 (5 voters).** Used by this repo's example and walkthroughs below.

**3-3-1 (7 voters)** is functionally fine if you want extra within-main-region headroom — majority=4 forces every commit to span ≥2 regions (per-DC voters in the main DCs = 3 < 4), same as 2DC 3-3. The cost: 4 acks per commit vs 3 for 2-2-1.

**Avoid 4-4-1 (9 voters)** and any larger main-region count. With majority=5, the tiebreaker's 1 voter contributes nothing toward making writes commit (you need 5 acks; the tiebreaker provides at most 1 of them, leaving 4 of 5 acks that must come from the 2 main DCs — but each main DC only has 4 voters, so you're already needing both main DCs to participate in every commit). The extra voters don't add fault-tolerance against a full-region loss (still f=4, dominated by losing a main DC's 4 voters), they just make every commit pay 5 acks instead of 3 (2-2-1) or 4 (3-3-1). On top of that, the increase from 5 → 9 voters means each commit's tail-latency exposure is the slowest-of-5 instead of slowest-of-3 — measurably worse with cross-region links.

- **Fault tolerance**: 5 voters, majority = 3, tolerates 2 voter failures.
- **`alive/needed/configured` walkthrough** — healthy 2.5DC is `5/3/5` (5 alive, need 3, 5 configured), headroom 2.

  Each row shows two transitions: what the failure did to the state, and what your action does on top of it.

  | Failure | What happened | What to do |
  |---|---|---|
  | None | `5/3/5` | — |
  | Lose the 0.5 region (tiebreaker, 1 voter) | `5/3/5 → 4/3/5` | **No need to shrink.** Shrinking → `4/3/4` keeps `needed=3`, so headroom stays at 1 either way. You're already in an effective 2DC rolling-restart situation the moment the tiebreaker dies; bring the tiebreaker back as soon as possible instead. |
  | Lose 1 main region (2 voters) | `5/3/5 → 3/3/5` | **Shrink the voter set** (`remove-controller` the 2 dead voters) → `3/2/3` with headroom 1. Cluster is healthy; rolling restarts work; normal operations resume. The asymmetric layout caveat only matters if you then lose a *second* region before the first comes back (beyond design ceiling). See the rebalance worked example below for the shrink-first vs grow-first ordering choice. |
  | Lose 1.5 regions (1 main region + tiebreaker, or both main regions) **simultaneously** | quorum lost | Beyond the topology's design ceiling (more than half of N=5 lost). DR via `force-standalone` is possible, but **data-loss risk is real** — commits could have been ack'd entirely within the dead regions. Worked DR for this case: [`disaster-recovery/quorum-loss-recovery/`](../disaster-recovery/quorum-loss-recovery/). |
  | **Sequential** failure (lose 1 region, properly handle it, then lose another) | depends on intermediate state | If the intermediate handling rebalanced to a symmetric cross-DC layout (e.g. `4/3/4` as `2-0-2`), every commit during that window was cross-DC-forced — the surviving DC has all writes, no data-loss risk. If the intermediate state was still asymmetric (the shrink-only `3/2/3`), the second failure inherits the data-loss-window risk from the first. Outcome depends on what you did between the failures. |

  **Why the two main-region cases differ on shrink mechanics**: shrinking only buys headroom when the new voter count drops `needed`.
  - Tiebreaker loss: pre-shrink `4/3/5`, post-shrink `4/3/4`. `needed` stays 3 (majority of 4 = 3, same as majority of 5). Headroom unchanged at 1.
  - Main-region loss: pre-shrink `3/3/5`, post-shrink `3/2/3`. `needed` drops 3→2 (majority of 3 = 2). Headroom goes from 0 to 1 — cluster is healthy. The 2-1 layout is asymmetric (majority fits inside the 2-voter DC), which would matter only if a *second* region also fails before the first comes back. See the rebalance worked example below.

  **Rebalance worked example — main-region loss (dc1 dies, surviving dc2 and dc3):**

  Both realistic recovery paths are 2-step rebalances that end at the same symmetric `4/3/4` state (`2-_-2` layout, dc3 grown to 2 voters). They differ only in the **order** of `remove-controller` (shrink the dead voters) vs `add-controller` (grow dc3).

  | Step | Layout | `alive/needed/configured` | Headroom | Cross-DC commit forced? |
  |---|---|---|---|---|
  | Start | 2-2-1 (dc2-dc1-dc3) | 5/3/5 | 2 | yes |
  | After failure | 2-_-1 | 3/3/5 | 0 | yes |
  | **Path A: shrink-first** | | | | |
  | A1. `remove-controller` the 2 dead voters | 2-_-1 | 3/2/3 | 1 | **no** (majority=2 fits inside the 2-voter dc2 — same hole as 2DC 2-3) |
  | A2. `add-controller` to grow dc3 | 2-_-2 | 4/3/4 | 1 | yes (majority=3 > per-DC voters) |
  | **Path B: grow-first** | | | | |
  | B1. `add-controller` to grow dc3 (dc1 dead voters still in set) | 2-_-2 | 4/4/6 | 0 | yes (majority=4 > per-DC voters) |
  | B2. `remove-controller` the 2 dead voters | 2-_-2 | 4/3/4 | 1 | yes |

  Both paths land at the same final state (`4/3/4`, cross-DC forced, headroom 1). They differ in what the cluster looks like between the two membership changes:

  - **Path A's intermediate (`3/2/3`)**: healthy cluster with headroom 1 (rolling restarts work, single voter failures absorbed). The asymmetric layout (majority=2 fits inside dc2) means that *if dc2 then fails before dc1 returns* — i.e. a second region loss — `force-standalone` on dc3 could be missing writes (RPO > 0). But losing a second region is already beyond 2.5DC's design ceiling (f=2 bound exceeded), so this caveat only matters in territory the topology wasn't designed for anyway.
  - **Path B's intermediate (`4/4/6`)**: headroom 0 (because `needed` jumped 3→4 when `configured` went 5→6), but cross-DC commit stays forced. Closes the beyond-design-ceiling data-loss window that Path A leaves open.

  Path B is theoretically nicer for the catastrophic "lose two regions" scenario. Path A is the realistic recovery for the normal "lose one region" scenario — it's a healthy end state, not a degraded one. Dynamic quorum is what makes either rebalance achievable; static would force a config-rollout-and-restart of every voter.

  > **What shrink-only recovery actually gives you.** Shrink-only (Path A, stopping at `3/2/3`) is **not a degraded fallback** — it's a real recovery that materially improves the cluster's state. Before shrinking, you're at `3/3/5` with headroom 0: a KRaft rolling restart or any single voter blip takes out quorum. After shrinking, you're at `3/2/3` with headroom 1: rolling restarts work, single voter failures are absorbed, normal operations resume. The cluster is functionally healthy.
  >
  > The asymmetric-durability caveat only matters if **another region also fails before dc1 returns** — i.e. you lose a *second* main region on top of the first. That's the 1.5-DC simultaneous-loss case, which is already beyond 2.5DC's design ceiling (it exceeds the f=2 tolerated-failures bound regardless of what voter layout you're in). Path B (grow-first) is theoretically nicer because it closes that beyond-design-ceiling data-loss window too — but the window only opens if you cross into territory the topology isn't designed for in the first place.
  >
  > **CFK support status**: Path B's `add-controller` step requires scaling a `KRaftController` CR's `replicas` from N to N+1 with the new pod auto-joining the dynamic quorum. **CFK does not cleanly support that today** — it needs manual intervention (or isn't reliably supported at all). The 2.5DC disaster-recovery example in [`disaster-recovery/no-quorum-loss-recovery/`](../disaster-recovery/no-quorum-loss-recovery/) does Path A (shrink-only), which is fully supported by stock CFK and produces a healthy `3/2/3` cluster.
- **Kafka data durability**: with Kafka brokers in only 2 regions, replicas can span at most 2 regions. Lose 1 region with `RF=3, min.isr=2`: ISR shrinks to 1, writes block until you lower min.isr or the region returns. **2.5DC's data durability story is no better than 2DC's**; the third region is for KRaft, not for Kafka data.
- **RTO**: zero for any 1-DC failure (the cluster stays available, no DR triggered). Non-zero once you cross into the 1.5-DC-loss case — same shape as 2DC's force-standalone time, depends on your runbook readiness. Rehearse to measure.
- **RPO**: zero on any **1-DC** failure (Raft's majority-intersection guarantee — majority=3, no region holds ≥3 voters, so every commit spans ≥2 regions and survivors have it). For the **1.5-DC simultaneous** case, the guarantee no longer applies — data loss is possible. See the failure-table row above.
- **DR sketch**:
  - **1-DC loss (main or tiebreaker)** — quorum maintained; no DR. Optionally rebalance the voter layout in surviving DCs (grow-before-shrink, see the rebalance worked example in the availability bullet above) to restore symmetric durability + headroom.
  - **1.5-DC simultaneous loss** — beyond design ceiling; DR is possible but **data-loss risk is real**. See the failure-table row above and [`disaster-recovery/quorum-loss-recovery/`](../disaster-recovery/quorum-loss-recovery/).

  This repo's example is the worked end-to-end DR for 2.5DC: [`disaster-recovery/quorum-loss-recovery/`](../disaster-recovery/quorum-loss-recovery/) (1.5-DC simultaneous loss) and [`disaster-recovery/no-quorum-loss-recovery/`](../disaster-recovery/no-quorum-loss-recovery/) (1-DC loss → rebalance).
